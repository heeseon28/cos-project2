# COS Project 2 참고 요약

## 핵심 흐름

이 프로젝트의 핵심은 Edge Device, Control Server, AI Module을 서로 연결해 전력 사용량 예측 흐름을 완성하는 것이다.

```text
Edge Device
-> process_manager.cpp에서 하루치 데이터를 aggregation
-> network_manager.cpp에서 byte payload를 Server로 전송
-> server.py에서 payload를 decoding
-> AI Module에 JSON으로 전달
-> training/testing/result 수행
```

세 파일은 반드시 한 세트로 맞아야 한다.

```text
process_manager.cpp : 어떤 값을 어떤 byte로 저장할지 결정
network_manager.cpp : 그 byte payload를 몇 byte 보낼지 결정
server.py            : 받은 byte를 어떤 순서와 크기로 해석할지 결정
```

## process_manager.cpp

### 역할

`process_manager.cpp`의 핵심 함수는 `processData()`이다.

`processData()`는 `DataSet`에서 온도, 습도, 전력, 날짜 정보를 꺼내 요약하고, Server로 보낼 byte buffer를 만든다.

```text
DataSet
-> feature 계산
-> byte buffer에 encoding
-> dlen에 payload 길이 기록
-> NetworkManager로 전달
```

### 현재 Starter 예시

현재 feature vector는 다음과 같다.

```text
[min_temp, min_humid, min_power, month]
```

현재 byte payload는 다음과 같다.

```text
min_temp  : 1 byte
min_humid : 1 byte
min_power : 2 bytes
month     : 1 byte
```

총 payload 길이:

```text
5 bytes
```

### 수정점

#### 1. 변수 선언부 수정

어떤 feature를 쓸지에 맞게 변수를 준비한다.

예시:

```cpp
int avg_power, max_temp, min_temp, avg_humid, month;
long long sum_power;
```

#### 2. Aggregation 계산부 수정

`DataSet`에서 원하는 값을 꺼내거나 직접 계산한다.

추천 5-feature 설계 예시:

```text
avg_power  : 모든 HouseData의 power 평균
max_temp   : tdata->getMax()
min_temp   : tdata->getMin()
avg_humid  : hdata->getValue()
month      : ds->getTimestamp()에서 localtime()으로 추출
```

#### 3. Buffer Encoding 및 dlen 수정

계산한 feature를 `byte_op.h`의 macro로 buffer에 저장한다.

4 bytes로 저장하려면 macro와 `dlen` 증가량이 반드시 같이 맞아야 한다.

```cpp
VAR_TO_MEM_4BYTES_BIG_ENDIAN(avg_power, p);
*dlen += 4;
```

잘못된 예:

```cpp
VAR_TO_MEM_1BYTE_BIG_ENDIAN(avg_power, p);
*dlen += 4;
```

위 코드는 실제로는 1 byte만 저장했는데, 길이는 4 bytes라고 기록하는 것이므로 protocol이 깨진다.

`dlen`은 저장 크기를 정하는 값이 아니라, 실제로 저장된 payload 길이를 `NetworkManager`에게 알려주는 값이다.

## network_manager.cpp

### 역할

`network_manager.cpp`는 Edge와 Server 사이의 TCP socket 통신을 담당한다.

주요 함수는 다음과 같다.

```text
init()           : Server에 TCP 연결
sendData()       : OPCODE_DATA와 payload 전송
receiveCommand() : Server의 DONE/WAIT/QUIT opcode 수신
```

### 현재 Starter 예시

현재 Edge -> Server packet 구조는 다음과 같다.

```text
OPCODE_DATA : 1 byte
payload     : 5 bytes
```

payload 5 bytes는 `process_manager.cpp`가 만든다.

```text
min_temp || min_humid || min_power || month
```

### 수정점

#### sendData()의 Payload 길이 수정

현재는 payload 길이가 5로 고정되어 있다.

```cpp
tbs = 5;
```

`processData()`의 payload 길이가 바뀌면 이 값도 바뀌어야 한다. 가장 좋은 방식은 `processData()`가 넘겨준 `dlen`을 그대로 사용하는 것이다.

```cpp
tbs = dlen;
```

또는 다음처럼 직접 `dlen` 기준으로 반복해도 된다.

```cpp
offset = 0;
while (offset < dlen)
{
  sent = write(sock, data + offset, dlen - offset);
  if (sent > 0)
    offset += sent;
}
```

### 유지할 부분

`OPCODE_DATA`를 먼저 보내는 부분은 유지한다.

```cpp
opcode = OPCODE_DATA;
```

이 opcode는 Server에게 "지금부터 data packet을 보낸다"는 의미를 알려준다.

`receiveCommand()`는 특별한 이유가 없다면 크게 수정하지 않아도 된다. Server가 `WAIT`을 보내면 기다리고, `DONE` 또는 `QUIT`이 오면 반환하는 구조이다.

## server.py

### 역할

`server.py`는 Edge와 AI Module 사이의 통역사 역할을 한다.

```text
Edge와 통신      : socket byte protocol
AI Module과 통신 : HTTP/JSON REST API
```

주요 함수는 다음과 같다.

```text
connecter()     : AI Module에 모델 생성 요청
listener()      : Edge 접속 대기
handler()       : training/testing 흐름 관리
parse_data()    : Edge payload decoding
send_instance() : AI Module에 JSON feature list 전달
print_result()  : 최종 prediction result 출력
```

### 현재 Starter 예시

현재 `handler()`는 payload를 5 bytes로 받는다.

```python
rbuf = client.recv(5)
```

현재 `parse_data()`는 5 bytes를 다음처럼 해석한다.

```text
temp  = buf[0:1]
humid = buf[1:2]
power = buf[2:4]
month = buf[4:5]
```

AI Module에 보내는 list:

```text
[temp, humid, power, month]
```

따라서 현재 starter 기준 실행 옵션은 다음과 같다.

```bash
--dimension 4 --index 2
```

### 수정점

#### 1. PAYLOAD_LEN 추가

opcode 정의 아래에 payload 길이를 상수로 두면 관리가 쉽다.

```python
OPCODE_DATA = 1
OPCODE_WAIT = 2
OPCODE_DONE = 3
OPCODE_QUIT = 4

PAYLOAD_LEN = 20
```

#### 2. parse_data() 수정

`process_manager.cpp`에서 저장한 byte 순서와 크기 그대로 decoding한다.

추천 5-feature protocol 예시:

```python
def parse_data(self, buf, is_training):
    avg_power = int.from_bytes(buf[0:4], byteorder="big", signed=True)
    max_temp = int.from_bytes(buf[4:8], byteorder="big", signed=True)
    min_temp = int.from_bytes(buf[8:12], byteorder="big", signed=True)
    avg_humid = int.from_bytes(buf[12:16], byteorder="big", signed=True)
    month = int.from_bytes(buf[16:20], byteorder="big", signed=True)

    lst = [avg_power, max_temp, min_temp, avg_humid, month]
    self.send_instance(lst, is_training)
```

#### 3. handler()의 Training 수신부 수정

현재:

```python
rbuf = client.recv(5)
```

수정 방향:

```python
rbuf = client.recv(PAYLOAD_LEN)
```

#### 4. handler()의 Testing 수신부 수정

현재:

```python
rbuf = client.recv(5)
```

수정 방향:

```python
rbuf = client.recv(PAYLOAD_LEN)
```

#### 5. 더 안전한 구현: recv_exact()

TCP의 `recv(20)`은 항상 20 bytes를 한 번에 돌려준다는 보장이 없다. 더 안전하게 하려면 정확한 byte 수가 모일 때까지 반복해서 받는 helper를 둔다.

```python
def recv_exact(self, client, size):
    data = b""
    while len(data) < size:
        chunk = client.recv(size - len(data))
        if not chunk:
            logging.error("[*] connection closed while receiving data")
            sys.exit(1)
        data += chunk
    return data
```

사용 예:

```python
rbuf = self.recv_exact(client, PAYLOAD_LEN)
```

## 추천 최종 설계 예시

### Feature Vector

```text
[avg_power, max_temp, min_temp, avg_humid, month]
```

### Aggregation 종류

| Feature | Aggregation / 의미 |
|---|---|
| `avg_power` | average, 예측 대상 전력 평균 |
| `max_temp` | max, 냉방 수요 관련 가능 |
| `min_temp` | min, 난방 수요 관련 가능 |
| `avg_humid` | average, 체감 온도 관련 가능 |
| `month` | time/season feature |

### Protocol

```text
OPCODE_DATA : 1 byte
avg_power   : 4 bytes signed big-endian
max_temp    : 4 bytes signed big-endian
min_temp    : 4 bytes signed big-endian
avg_humid   : 4 bytes signed big-endian
month       : 4 bytes signed big-endian
```

payload 길이:

```text
20 bytes
```

packet 전체 길이:

```text
21 bytes
```

server 실행 옵션:

```bash
--dimension 5 --index 0
```

이유:

```text
feature가 총 5개이므로 dimension은 5이다.
power 값인 avg_power가 feature vector의 0번째에 있으므로 index는 0이다.
```

## 수정할 때 반드시 맞춰야 하는 것

1. `process_manager.cpp`에서 buffer에 저장하는 feature 순서
2. `process_manager.cpp`에서 각 feature를 저장하는 byte 크기
3. `process_manager.cpp`의 `*dlen` 값
4. `network_manager.cpp`가 보내는 payload 길이
5. `server.py`가 받는 payload 길이
6. `server.py`의 `parse_data()` slicing 순서
7. server 실행 옵션 `dimension` / `index`

하나라도 어긋나면 값이 이상하게 decoding되거나 Server가 payload를 기다리다 멈출 수 있다.

## 보고서에 쓸 수 있는 설명

우리는 전력 사용량 예측을 위해 평균 전력, 최고 온도, 최저 온도, 평균 습도, 월을 feature로 선택하였다. 평균 전력은 예측 대상이고, 최고/최저 온도는 냉난방 수요와 관련될 수 있다. 평균 습도는 체감 온도에 영향을 줄 수 있으며, 월은 계절성을 반영한다.

protocol은 모든 feature를 4 bytes signed big-endian integer로 encoding하도록 설계하였다. 이 방식은 음수 온도와 큰 전력값을 안정적으로 표현할 수 있고, C++ Edge와 Python Server 사이에서 byte slicing을 단순하게 유지할 수 있다.
